## Technical prompt

Convert Fixit 24/7 into a production-grade **multi-role marketplace app** with three role experiences: **Admin**, **Homeowner**, and **Tradie**. Build it as a real app, not a marketing site. Use a scalable architecture with PostgreSQL, role-based auth, modular routes, reusable components, mobile-first UX, and strong conversion-focused product flows. The codebase should validate only the required environment variables and disregard everything else, using a centralized config layer rather than scattering `process.env` checks across the app. Centralized env validation and modular Express routing improve maintainability and reduce configuration errors. [github](https://github.com/orgs/community/discussions/157077)

### Required environment variables only

Use only these required env variables:
- `APP_URL`
- `DATABASE_URL`
- `FROM_EMAIL`
- `NODE_ENV`
- `PORT`
- `SENDGRID_API_KEY`

- `SENDGRID_OTP_TEMPLATE_ID`
- `SENDGRID_TRADIE_WELCOME_TEMPLATE_ID`
- `SESSION_SECRET`

Ignore all other env variables if present. Build a config module that:

This matches common dotenv best practices for secure and portable Node deployments. [coreui](https://coreui.io/answers/how-to-use-dotenv-in-nodejs/)

### Core architecture

Build a single application with one account per email, but allow a single user to hold multiple roles at the same time. Use one unified identity with an active-role context in session/app state. After login:
- if user has one role, enter directly
- if user has multiple roles, show a “Continue as…” role chooser
- allow instant role switching from account menu without logout
- change permissions, navigation, and dashboard by active role
Modular route/controller structure in Express and a relational Postgres schema should be used for maintainability and role-aware access control. [itnext](https://itnext.io/building-restful-api-with-node-js-express-js-and-postgresql-the-right-way-b2e718ad1c66)

### Database schema

Create PostgreSQL schema and migrations for the following entities:

**Auth and users**
- `users`: id, full_name, email (unique), phone, password_hash or auth_provider fields, avatar_url, is_verified, created_at, updated_at
- `roles`: id, key (`admin`, `homeowner`, `tradie`), label
- `user_roles`: id, user_id, role_id, is_active, granted_by, granted_at, metadata_json
- `sessions` or session store table if needed

**Role profiles**
- `homeowner_profiles`: id, user_id, default_address_id, property_preferences_json
- `tradie_profiles`: id, user_id, business_name, abn, license_number, insured, verification_status, rating_avg, jobs_completed, service_radius_km
- `admin_profiles`: id, user_id, admin_level, permissions_json

**Addresses and service locations**
- `addresses`: id, user_id, label, line1, line2, suburb, state, postcode, lat, lng, is_default

**Jobs marketplace**
- `job_categories`: id, name, slug, icon
- `jobs`: id, homeowner_user_id, address_id, category_id, title, description, urgency, status, preferred_date, budget_min, budget_max, created_at, updated_at
- `job_photos`: id, job_id, image_url, sort_order
- `job_invites`: id, job_id, tradie_user_id, invited_at, status
- `quotes`: id, job_id, tradie_user_id, amount, message, estimated_start_date, status, created_at
- `job_assignments`: id, job_id, tradie_user_id, accepted_quote_id, assigned_at, status

**Messaging and notifications**
- `conversations`: id, job_id, homeowner_user_id, tradie_user_id, created_at
- `messages`: id, conversation_id, sender_user_id, message_type, body, attachment_url, read_at, created_at
- `notifications`: id, user_id, type, title, body, cta_url, is_read, created_at

**Payments and earnings**
- `payments`: id, job_id, payer_user_id, payee_user_id, amount, currency, status, provider_ref, paid_at
- `payouts`: id, tradie_user_id, amount, status, requested_at, paid_at

**Trust and moderation**
- `reviews`: id, job_id, reviewer_user_id, reviewee_user_id, rating, comment, created_at
- `documents`: id, tradie_user_id, type, file_url, verification_status, uploaded_at
- `disputes`: id, job_id, opened_by_user_id, reason, status, resolution_notes, assigned_admin_user_id, created_at, resolved_at
- `audit_logs`: id, actor_user_id, entity_type, entity_id, action, metadata_json, created_at

Add indexes, foreign keys, cascading rules, status enums, and seed data for roles and base categories. The schema should support future expansion cleanly. Structured schemas and clearly defined responsibilities make marketplace apps easier to evolve. [docsbot](https://docsbot.ai/prompts/technical/ai-prompt-marketplace-architecture-plan)

### UX behavior by role

Make each role meaningfully different:
- Homeowner UI should be trust-first, low-stress, and quote comparison centered.
- Tradie UI should be action-first, fast, and optimized for job response speed.
- Admin UI should be dense, scannable, filterable, and resolution-oriented.
Role-based marketplaces convert better when the product surfaces the user’s most important next action instead of showing everyone the same dashboard. [excited](https://excited.agency/blog/marketplace-ux-design)
